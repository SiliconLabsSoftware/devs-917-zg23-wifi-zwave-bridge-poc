Detailed Description

Control of the Z-Wave network is carried out through a number of Network Management Command Classes.

The Z/IP Gateway MUST support all four Network Management Command Classes:

  • Network Management Inclusion: Support for Adding/Removing, Set Default, Failed Node Replacement and Removal and Requesting Node Neighbor Update
  • Network Management Basic: Support for entering Learn Mode, Node Information Send and Request Network Update
  • Network Management Proxy: Support for Node List Get and Node Info Cached Get
  • Network Management Primary: Support for Controller Change, only available when operating as Primary Controller without SIS functionality.

These four command classes enable network management and access to simple network topology information.

◆ nm_state_t

enum nm_state_t

State of the Network Management module.

If the Network Management State machine (NMS) is not idle, Z/IP Gateway will reply COMMAND_BUSY to all Network Management commands.

Enumerator
NM_IDLE 

Idle state.

NM_WAITING_FOR_ADD 

Add mode state. Classic, S2, or Smart Start.

Waiting for protocol. On NM_EV_ADD_NODE_FOUND, go to NM_NODE_FOUND.

NM_NODE_FOUND 

Add mode state. Classic, S2, Smart Start, Proxy Inclusion, or Proxy Replace.

Waiting for protocol to receive node info from a node. On NM_EV_ADD_CONTROLLER or NM_EV_ADD_END_NODE, cache the node id and node info and go to NM_WAIT_FOR_PROTOCOL.

NM_WAIT_FOR_PROTOCOL 

Add mode state. Classic, S2, Smart Start, Proxy Inclusion, or Proxy Replace.

Waiting for protocol to assign nodeid and homeid to the new node.

On NM_EV_ADD_PROTOCOL_DONE, send an ADD_NODE_STOP to protocol. On NM_EV_ADD_NODE_STATUS_DONE, register the new node in Resource Directory Node Database and go to NM_WAIT_FOR_SECURE_ADD. If GW is handling inclusion, start security processing the new node (S2 or S0 inclusion). Otherwise, start a 2 s timer and go to NM_PREPARE_SUC_INCLISION.

NM_NETWORK_UPDATE 

GW is processing NETWORK_UPDATE_REQUEST.

GW is not SUC/SIS and is processing a NETWORK_UPDATE_REQUEST.

ResetState() from nm_send_reply() will bring NMS back to NM_IDLE.

NM_WAITING_FOR_PROBE 

Learn mode state. Also used if GW is processing NODE_INFO_CACHED_GET, NETWORK_UPDATE_REQUEST, or DEFAULT_SET.

Waiting on networkUpdateStatusFlags. When all flags are in, ResetState() from nm_send_reply() will bring NMS back to NM_IDLE.

Cannot be interrupted.

NM_SET_DEFAULT 

Learn mode state (exclusion). Also used if GW is processing DEFAULT_SET.

Waiting for NetworkManagement_mdns_exited() to be called from ZIP_Router. Post ZIP_EVENT_RESET to ZIP_Router and call SendReplyWhenNetworkIsUpdated() to go to NM_WAITING_FOR_PROBE. SendReplyWhenNetworkIsUpdated() sets up a trigger to bring NMS back to NM_IDLE when the gateway reset is completed.

Note
When changing to this state, rd_exit() MUST be called to advance the state machine.
NM_LEARN_MODE 

Learn mode state.

GW is processing LEARN_SET, waiting for protocol.

On callback (LearnModeStatus()) with LEARN_MODE_STARTED, go to NM_LEARN_MODE_STARTED, Lock RD probe machine, invalidate MyNodeID, and start S2 learn-mode state machine.

Can be canceled or time out. This will trigger a partial re-initialization of the gateway (sl_application_nif_init()), unlock RD probe. ResetState() from nm_send_reply() with the LEARN_MODE_FAILED will bring NMS back to NM_IDLE.

NM_LEARN_MODE_STARTED 

Learn mode state.

The gateway is processing LEARN_SET and has received LEARN_MODE_STARTED from protocol. Now, the protocol is committed and the gateway waits for protocol completion.

On protocol callback (LearnModeStatus()) with LEARN_MODE_DONE, the gateway has been assigned a nodeid and homeid. Go to NM_WAIT_FOR_SECURE_LEARN, detect new environment, and reset the gateway accordingly.

Cannot be interrupted because the gateway properties have changed and the wait is for the protocol.

NM_WAIT_FOR_SECURE_ADD 

Add mode state. Classic, S2, Smart Start, Proxy Inclusion, Proxy Replace.

Waiting for the S2 inclusion state machine to complete. On NM_EV_SECURITY_DONE, go to NM_WAIT_FOR_PROBE_AFTER_ADD and start probe.

If Smart Start security fails, start a SMART_START_SELF_DESTRUCT_TIMEOUT sec. timer and go to NM_WAIT_FOR_SELF_DESTRUCT.

NM_SENDING_NODE_INFO 

GW is processing NODE_INFORMATION_SEND.

__ResetState() callback from protocol will bring NMS back to NM_IDLE.

NM_WAITING_FOR_NODE_REMOVAL 

Remove node state.

Waiting for protocol. In RemoveNodeStatusUpdate() callback on REMOVE_NODE_STATUS_DONE status, remove the node from Resource Directory, start removing the node's ip associations, and go to NM_REMOVING_ASSOCIATIONS. On REMOVE_NODE_STATUS_FAILED, ResetState() from nm_send_reply() will bring NMS back to NM_IDLE.

NM_WAITING_FOR_FAIL_NODE_REMOVAL 

Remove failed node state.

Waiting for protocol. In callback RemoveFailedNodeStatus(), ResetState() from nm_send_reply() will bring NMS back to NM_IDLE.

NM_WAITING_FOR_NODE_NEIGH_UPDATE 

GW is processing NODE_NEIGHBOR_UPDATE_REQUEST.

When receiving REQUEST_NEIGHBOR_UPDATE_DONE or REQUEST_NEIGHBOR_UPDATE_DONE, ResetState() from nm_send_reply() will bring NMS back to NM_IDLE.

NM_WAITING_FOR_RETURN_ROUTE_ASSIGN 

GW is processing RETURN_ROUTE_ASSIGN.

Waiting for protocol. In callback AssignReturnRouteStatus(), ResetState() from nm_send_reply() will bring NMS back to NM_IDLE.

NM_WAITING_FOR_RETURN_ROUTE_DELETE 

GW is processing RETURN_ROUTE_DELETE.

Waiting for protocol. In callback DeleteReturnRouteStatus(), ResetState() from nm_send_reply() will bring NMS back to NM_IDLE.

NM_WAIT_FOR_PROBE_AFTER_ADD 

Add mode state. Classic, S2, Smart Start, Replace Failed, Proxy Inclusion, Proxy Replace.

Depends on ZIP_Router to call NetworkManagement_node_probed() to advance NMS to NM_WAIT_DHCP.

NM_WAIT_FOR_SECURE_LEARN 

Learn mode state.

The gateway has received LEARN_MODE_DONE from protocol.

For inclusion, Network Management is waiting for S2/S0 learn mode to complete. On NM_EV_SECURITY_DONE, go to NM_WAIT_FOR_MDNS and call rd_exit() to reset the gateway for a new network.

For NM_EV_ADD_SECURITY_KEY_CHALLENGE, accept. For NM_EV_LEARN_SET/DISABLE, abort S2.

For exclusion, controller replication, and controller shift, NM_EV_SECURITY_DONE is triggered synchronously in NMS and state is changed synchronously when LEARN_MODE_DONE is received to either NM_SET_DEFAULT or NM_WAIT_FOR_MDNS.

NM_WAIT_FOR_MDNS 

Learn mode state.

After NM_WAIT_FOR_SECURE_LEARN, NMS is waiting for NetworkManagement_mdns_exited() to be called from ZIP_Router. For inclusion or exclusion, NMS then calls bridge_reset().

If NMS_FLAG_LEARNMODE_NEW is set, send the LEARN_MODE_SET_STATUS reply, go to NM_WAIT_FOR_PROBE_BY_SIS and set up a 6 s timer. Otherwise, post ZIP_EVENT_RESET to ZIP_Router and call SendReplyWhenNetworkIsUpdated() to go to NM_WAITING_FOR_PROBE. SendReplyWhenNetworkIsUpdated() sets up a trigger to bring NMS back to NM_IDLE when the gateway reset is completed.

Note
When changing to this state, rd_exit() MUST be called to advance the state machine.
NM_WAIT_FOR_PROBE_BY_SIS 

Learn mode state. Only used with NMS_FLAG_LEARNMODE_NEW.

After learn mode, the gateway must wait for the SIS to probe before resetting. As a result, stay in NM_WAIT_FOR_PROBE_BY_SIS and restart the 6 s timer every time a frame is received.

On timeout, i.e., when no frames have been received for at least 6 s, the assumption is that SIS is done. Post ZIP_EVENT_RESET to ZIP_Router and go to NM_WAIT_FOR_OUR_PROBE.

NM_WAIT_FOR_OUR_PROBE 

Learn mode state.

Only used with NMS_FLAG_LEARNMODE_NEW.

Wait for NM_EV_ALL_PROBED, then call SendReplyWhenNetworkIsUpdated() to go to NM_WAITING_FOR_PROBE. SendReplyWhenNetworkIsUpdated() sets up a trigger to bring NMS back to NM_IDLE when the gateway reset is completed.

NM_WAIT_DHCP 

Add mode state.

Classic, S2, Smart Start, Proxy Inclusion, Proxy Replace.

Wait for NM_EV_DHCP_DONE, when the new node has an IPv4 address, or timeout (5s timer). Then call nm_send_reply(), where the callback ResetState() will bring NMS back to NM_IDLE. For smart start, call ResetState() directly to bring NMS back to NM_IDLE.

NM_REMOVING_ASSOCIATIONS 

Remove node state.

Waiting for bridge. Callback sl_nm_virtual_nodes_removed() will bring NMS back to NM_IDLE.

NM_REPLACE_FAILED_REQ 

Replace failed state.

Replace and replace S2.

On NM_EV_REPLACE_FAILED_DONE, if GW is handling inclusion, go to NM_WAIT_FOR_SECURE_ADD. Otherwise, go to NM_PREPARE_SUC_INCLISION.

NM_PREPARE_SUC_INCLISION 

Add mode state.

Classic, S2, Smart Start, Replace Failed, Proxy Inclusion, or Proxy Replace.

Wait for timeout to let protocol finish up its add by sending a transfer end to SUC. Then, go to NM_WIAT_FOR_SUC_INCLUSION and request handover.

NM_WIAT_FOR_SUC_INCLUSION 

Add mode state.

Classic, S2, Smart Start, Replace Failed, Proxy Inclusion, or Proxy Replace.

Waiting for the SUC to complete the secure part of inclusion.

On NM_EV_PROXY_COMPLETE, go to NM_WAIT_FOR_SECURE_ADD and post NM_EV_SECURITY_DONE.

NM_PROXY_INCLUSION_WAIT_NIF 

Add mode state. Proxy inclusion and proxy replace.

Safe-guarded by a timer, on timeout, go to NM_IDLE.

On NM_EV_NODE_INFO, inclusion goes to NM_NODE_FOUND and fakes NM_EV_ADD_CONTROLLER and NM_EV_ADD_NODE_STATUS_DONE. Replace goes to NM_REPLACE_FAILED_REQ and fakes NM_EV_REPLACE_FAILED_DONE.

NM_WAIT_FOR_SELF_DESTRUCT 

Add mode state. Smart Start.

Smart Start inclusion has failed and gateway is waiting for the node to self destruct.

Wait for timeout, then send a NOP and go to NM_WAIT_FOR_TX_TO_SELF_DESTRUCT.

NM_WAIT_FOR_SELF_DESTRUCT_RETRY 

Add mode state. Smart Start.

First attempt of self destruct has failed and gateway is waiting for the node to to retry the self destruct.

Wait for timeout, then send a NOP and go to NM_WAIT_FOR_TX_TO_SELF_DESTRUCT_RETRY.

NM_WAIT_FOR_TX_TO_SELF_DESTRUCT 

Add mode state. Smart Start.

The gateway must send NOPs to a failing node and wait for the result, before doing ZW_RemoveFailed().

The NOP is sent before entering this state. Now the gateway waits for protocol callback nop_send_done() to trigger NM_EV_TX_DONE_SELF_DESTRUCT. Then we start a timer, call ZW_RemoveFailedNode() and go to NM_WAIT_FOR_SELF_DESTRUCT_REMOVAL.

NM_WAIT_FOR_TX_TO_SELF_DESTRUCT_RETRY 

Add mode state. Smart Start.

The gateway must retry sending NOPs to a failing node and wait for the result, before doing ZW_RemoveFailed().

The NOP is sent before entering this state. Now, the gateway waits for protocol callback nop_send_done() to trigger NM_EV_TX_DONE_SELF_DESTRUCT_RETRY. Then, a timer is started, call ZW_RemoveFailedNode() and go to NM_WAIT_FOR_SELF_DESTRUCT_REMOVAL.

NM_WAIT_FOR_SELF_DESTRUCT_REMOVAL 

Add mode state. Smart Start.

Waiting for protocol to complete the ZW_RemoveFailedNode() (or for timeout). Protocol callback RemoveSelfDestructStatus() triggers sending status to unsolicited destinations with #ResetState() callback. Timeout calls #ResetState() directly.

NM_WAIT_FOR_SELF_DESTRUCT_REMOVAL_RETRY 

Add mode state. Smart Start.

Waiting for protocol to complete the retry of ZW_RemoveFailedNode() (or for timeout). Protocol callback RemoveSelfDestructStatus() triggers sending status to unsolicited destinations with #ResetState() callback. Timeout calls #ResetState() directly.

NM_WAIT_FOR_NODE_INFO_PROBE 

NodeInfoCachedGet state.

Used when Z/IP Gateway needs to probe because the cached info is too old. When the interview is completed, ZIP_Router will trigger NMS again by calling NetworkManagement_node_probed().

NM_WAIT_FOR_NEIGHBOR_UPDATE_AFTER_SECURE_ADD 

Used when Z/IP Gateway needs to wait for neighbor update after s2 inclusion because the neighbor update was skipped by protocol because of add node was requested with "Skip FL nodes in Neighbors Discovery"

◆ sl_nm_init()

int sl_nm_init ( )

Initialize this module, and, if possible, send a message from the previous life of the gateway.

Some network management commands will cause the Z/IP gateway to reset, i.e., it will restart its IPv6 stack, cancel all pending processes, and reprobe the network. This does all take time. The IP reply to the command should only be sent when all the probing is done.

Along with the boot process, sl_nm_init will be called several times. Each time it will check if it is able to send the pending IP package. If the package is sent, this function will return 1. Otherwise, it will return 0.

If network_management_init_done == 1, calling this function will have no effect. If init is successful, network_management_init_done will be set to 1.

Returns
TRUE on success.

Return 0 when init is pending on DHCP assignment, 1 when init is complete

◆ sl_nm_net_update_status()

void sl_nm_net_update_status ( uint8_t flag)

Notify the network management module that some event has occurred.

Should be called when ZIP_Router is told that all nodes have IPV4 addresses, that the bridge is up, that all nodes have been probed, or when NMS times out while waiting for network update to complete.

Determines the gateway status and calls nm_send_reply() when all requirements are fulfilled.

The function will test for the gateway status, but it will always assume that the event indicated by flag has happened, to ensure that network management does not get stuck.

Parameters
flagMask indicating which events have completed. Will be OR'd to reality.
